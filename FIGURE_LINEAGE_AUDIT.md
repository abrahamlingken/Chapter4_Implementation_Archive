# Chapter 4 Figure Lineage Audit

Scope: `figures/chapter4/` in this archive project.

## Core Findings

- The current `figures/chapter4/` outputs are generated by `Current/Evaluation/generate_chapter4_thesis_figures.py`, not by `Legacy_Figures3D/plot_3d_paper_figures.py`.
- The generation script is present in this project.
- The training entry script `Current/Training/train_3d_multicase.py` is present in this project.
- The model-dependent training output folders `Results_3D_CaseA/`, `Results_3D_CaseB/`, and `Results_3D_CaseC/` are not included inside this archive project.
- The corresponding model and training-history files do still exist outside this project in the parent workspace:
  - `../Results_3D_CaseA/model.pkl`
  - `../Results_3D_CaseA/training_history.json`
  - `../Results_3D_CaseA/case_config.json`
  - `../Results_3D_CaseB/model.pkl`
  - `../Results_3D_CaseB/training_history.json`
  - `../Results_3D_CaseB/case_config.json`
  - `../Results_3D_CaseC/model.pkl`
  - `../Results_3D_CaseC/training_history.json`
  - `../Results_3D_CaseC/case_config.json`
- Case A thesis figures do not load `model.pkl` directly in the current generator. They reuse archived validation data from `Artifacts/Figures_3D_Validation/`.
- Case B and Case C thesis figures depend on both archived FMC benchmark data under `Artifacts/MC3D_Raw_Benchmarks/` and external `Results_3D_Case{B,C}/model.pkl`.
- `Fig4-7` only uses the Case C model in `figure_fig4_7(...)`, but the top-level `run()` currently still resolves Case C FMC as a precondition, so the runtime dependency is stricter than the figure function itself.

- Split point 1: `figures/chapter4/` is inside the archive project, but `Results_3D_CaseA/B/C/` is outside it.
- Split point 2: Case A thesis figures were regenerated from archived validation `.npz`, not directly from raw training outputs.
- Split point 3: The project stores the current archive commit, but not the exact historical Git commit that produced the `2026-04-13` figure set.
